Java Linter

A linter is a programming 'spell checker' that most IDEs run their own versions of. Our linter is able to identify some cursory violations, OO principle violations, and identify modular design patterns.

In a team of 3, we designed and implemented a flexible framework using different Object-Oriented design patterns. The Java ASM library was used to analyze inputted Java Projects by default.

Our Design

Cursory Style Checks

  1. Poor naming convention: naming of objects should follow the formal naming conventions as listed in: Oracle's Naming Convention Page
  2. Equals HashCode: will flag any classes that override equals without also overriding hashCode or vice-versa. This is to ensure hashtable functionality.
  3. No Finalizer: checks that no classes contain a finalize method with zero parameters. CheckStyle's Info on Finalizers

Principle Violation Checks

  1. Information Hiding: will flag getters/setters as public methods that return a private field without further computation (will not be flagged in dataclasses as they are okay)
  2. Hollywood Principle: will flag low level components which directly call high level components
  3. Single Responsibility Principle: will flag classes with above the median amount of public methods (or user can specify)

Modular Design Pattern Identifiers

  1. Singleton Pattern: identifies proper usage of either 'lazy' or 'eager' Singleton Patterns defined when a class has a static private instance variable of itself, a constructor of private access, and has a static public method that returns itself
  2. Strategy Pattern: flags abstract classes (or interfaces) that has at least one concrete implementation.
  3. Decorator Pattern: flags an abstract class that inherits another abstract class and also holds a instance of said abstract class

Interpreter Project

The focus of this project was to create a interpreter in Scheme for my Programming Language Concepts class at RHIT. Additional Scheme specific datatyping/continuation concepts were learned and applied.

The Interpreting Process

  1. Scheme expressions were first parsed into expressions that were more suitable for transformation.
  2. These abstract expressions were then expanded to more primitive forms that could be evaluated by primitive expressions such as + - * / cons list? number? symbol? etc.
  3. Then the expression would be evaluated directly. Closures were created for lambdas and environments for variables.
